  ---
  두 개 앵커포인트 방식

  좋은 접근이야. 근데 우선순위 전환 상황을 생각해봐.

  [현재] 돈 3개 → Row1 앵커에 쌓여있음
  [광물 들어옴] → 돈 전체가 Row2 앵커로 이동해야 함

  돈이 Row1 → Row2로 이동할 때 모든 돈 아이템 위치를 다시 계산해야 해. 관리가
  약간 복잡해질 수 있어.

  ---
  자식으로 보내는 방식

  스프링 애니메이션(앞뒤 흔들림)이 나중 우선순위라면 지금은 괜찮아.

  근데 나중에 스프링 효과 넣을 때 자식 방식이 걸림돌이 돼 — 자식은 플레이어
  회전에 딱 붙어서 같이 돌아버리기 때문에 흔들리는 효과를 주기 어렵거든.

  ---
  그러면 고민해볼 것

  아이템 위치를 어떻게 계산할지야.

  [플레이어] → [광물0] → [광물1] → [광물2]
                                        ↓
                                [돈0] → [돈1]

  각 아이템이 자기 앞 아이템의 위치를 기준으로 offset을 두는 방식을 생각해봐.

  - Row1: 플레이어 바로 뒤부터 스택
  - Row2: Row1 마지막 아이템 뒤부터 스택? 아니면 Row1 옆(다른 축)?

  Row2는 Row1 기준으로 어느 방향에 쌓일지 결정해봐 — 이게 앵커 설계의 핵심이야.

  ---
  접근 자체는 좋아

  pos1/pos2를 플레이어 자식으로 두고 localPosition을 기준점으로 쓰는 건 맞는
  방향이야.

  핵심은 아이템 위치 계산할 때 이렇게 되는 거잖아:

  아이템 위치 = anchor.position + Vector3.up * (index * 간격)

  anchor가 플레이어 자식이라 플레이어가 회전해도 자동으로 따라가고.

  ---
  근데 한 가지 생각해볼 것

  우선순위 전환 시점이야.

  [현재] 돈 5개 → pos1에 쌓여있음
  [광물 들어옴] → 돈 5개 전부 pos2로 이동 + 광물은 pos1으로

  돈이 pos1 → pos2로 옮겨질 때 5개 전부 위치 재계산해야 해.

  이걸 어떻게 처리할지가 설계의 핵심 포인트야. 두 가지 방향이 있어:

  1. 전환 시 전체 재배치 — 단순하지만 아이템 많으면 한 번에 다 움직임
  2. 항상 광물 스택 / 돈 스택 분리 관리 — 각자 독립적으로 계산, 전환 시 anchor만
   바꿔치기

  어떤 방향이 더 자연스러울 것 같아?
